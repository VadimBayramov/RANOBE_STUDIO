> Этот документ — конституция проекта. Не руководство "хочется". Не список пожеланий.
> Каждое правило существует потому, что его нарушение разрушает систему на масштабе 500 глав.
> Если ты (автор или Claude) хочешь нарушить правило — сначала докажи, что это не регресс.

### 0.1 — Clarify Before Act

**Перед любым действием задаётся три вопроса:**
1. Что именно просит автор?
2. Зачем это нужно? (какую проблему решает)
3. Совпадает ли это с текущим состоянием документации и кода проекта (поиск через RLM)?

Если ответа на любой из вопросов нет — уточнить. Не догадываться. Не интерпретировать в пользу новой фичи.

### 0.2 — Несоответствие системным документам — БЛОК

Если запрос автора **противоречит** зафиксированному решению в документации текущего проекта, то
→ Claude **явно показывает несоответствие** и останавливается. Не продолжает. Не «делает как попросили». Не переформулирует запрос так, чтобы он «вписался». Сначала выравнивается документация - источник истины и после уже вносится правка в код.

Формат ответа:
```
⛔ НЕСООТВЕТСТВИЕ: [что именно противоречит] → [ссылка на источник]
Текущее решение: [что зафиксировано]
Твой запрос: [что ты просишь]
Варианты: (A) изменить запрос | (B) обновить системный документ через ADR
```

### 0.3 — Биохимический организм

Автор — человек. Сегодня он хочет одно, завтра другое. Claude — гарант долгосрочных решений.
Если автор **изобретает новую подсистему**, не связанную с текущим milestone, Claude:
1. Называет это прямо: «это out-of-scope для текущей фазы»
2. Оценивает, нужно ли это вообще (через призму горизонта — см. [6.1](#61--горизонт-планирования-500-глав))
3. Предлагает поместить в `backlog.md` — не реализует немедленно

**Исключение:** критические баги, блокирующие pipeline — чинятся немедленно.

### 0.4 — RLM как первый инструмент поиска

Перед тем как отвечать на вопрос о Knowledge Base, Canon, Research материалах — запустить:
```
/rlm "вопрос" --corpus KNOWLEDGE_BASE/cards/
```
Не пересказывать по памяти. Не угадывать. Если RLM не запущен — прямо сказать «я не проверил KB».
Полный протокол работы с KB — [Раздел 4](#раздел-4).

### 0.5 — Навигационная целостность документов

**Каждый документ проекта обязан содержать:**
1. **Шапку** с `related:` — список связанных файлов с кратким описанием роли
2. **Навигационную таблицу** — оглавление разделов со ссылками-якорями внутри документа
3. **Таблицу связанных документов** — кликабельные ссылки на все файлы из `related:`
4. **Обратные ссылки** `↑ [К навигации]` в начале каждого раздела

Документ без навигации = документ который не читают.

**Проверка:** если новый файл создаётся без этих четырёх элементов — это нарушение. Claude указывает на это до того как файл попадёт в pipeline.

**Обновление:** при добавлении нового связанного документа в проект — обновить `related:` и навигационные таблицы во всех файлах, которые на него ссылаются или должны ссылаться.

### 0.6 — Актуальность

Все рекомендации, технические решения, ссылки на инструменты, API и best practices —
должны быть актуальны на **апрель 2026** года. Устаревшие подходы называются устаревшими. Не применяются «по инерции».

## РАЗДЕЛ 1 — АРХИТЕКТУРНЫЕ ИНВАРИАНТЫ {#раздел-1}

*↑ [К навигации](#навигация-по-документу)*

### 1.1 — Модульность: файлы ссылаются, не встраиваются

Каждый файл системы независим и ссылается на другие по пути, не копирует их содержимое.

**Нарушение:** `scene_plan_ch_45.md` содержит полный текст `canon.md`.
**Правильно:** `scene_plan_ch_45.md` содержит `[world_rules] → см. [bible/canon.md](../01_bible/canon.md)`.

Принцип React-компонентов: каждый модуль знает свои входы и выходы, не знает о внутренней структуре других.

### 1.2 — Файловая система — единственный источник истины

Нет баз данных, нет in-memory state, нет «договорённостей». Всё фиксируется файлами:
- `bible/` — фундамент проекта → детали в [GENERATION_PIPELINE.md §Шаг 0](GENERATION_PIPELINE.md)
- `state_ch_N.yaml` — текущее состояние мира
- `loops.md` — нарративные петли
- `canon.md` — неизменяемые факты (правила изменения — [2.2](#22--canon-изменяется-только-явным-решением))
- `voice_corrections.md` — история правок голоса

Если чего-то нет в файлах — этого не существует.

### 1.3 — Навигация: папочная иерархия по workflow

Структура папок отражает этапы работы, нумерованные в порядке исполнения:
```
0. LIBRARY/        ← библиотека знаний (→ 0. LIBRARY/INDEX.md)
1. DESIGN_IDEA/    ← Слой 1: дизайн идеи
  1.1 PROMPT_IDEA/ ← шаблон + опросник (шаг 1.1)
  1.2 KB_ANALYSIS/ ← отчёт анализа идеи (шаг 1.2)
  1.3 BIBLE/       ← фундамент проекта (шаг 1.3)
  1.4 VOICE/       ← голос автора (шаг 1.4)
2. PLANNING/       ← Слой 2: арка (→ GENERATION_PIPELINE.md §Слой 2)
3. GENERATION/     ← Слой 3: генерация глав
  scene_plans/     ← планы сцен (шаг 3.1)
  drafts/          ← черновики (шаг 3.2)
  critiques/       ← отзывы критиков А, Б, В (шаг 3.3)
scripts/           ← build-скрипты (→ scripts/README.md)
.planning/         ← GSD артефакты
```

При ветвлении → подпапка. Никаких плоских `ch_001_scene_plan_draft_v2_final.md` в корне.

### 1.4 — Документы всегда синхронны с кодом

[GENERATION_PIPELINE.md](GENERATION_PIPELINE.md) описывает **фактический** pipeline, не желаемый.
Если pipeline изменился — документ обновляется в том же коммите.
Расхождение «доки говорят одно, скрипты делают другое» — критический баг, исправляется немедленно.

## РАЗДЕЛ 2 — ИНВАРИАНТЫ WORKFLOW {#раздел-2}

*↑ [К навигации](#навигация-по-документу)*

Полная схема workflow — [GENERATION_PIPELINE.md](GENERATION_PIPELINE.md). Этот раздел фиксирует инварианты, нарушение которых разрушает масштабируемость.

### 2.1 — Контекст генерации главы: ровно 5 файлов

```
voice_author.md    (cached)   → 1. DESIGN_IDEA/1.4 VOICE/
voice_pov.md       (cached)   → 1. DESIGN_IDEA/1.4 VOICE/
genre_overlay.md   (cached)   → 1. DESIGN_IDEA/1.4 VOICE/
scene_plan_ch_N.md (fresh)    → 3. GENERATION/scene_plans/
state_ch_N.yaml    (fresh)    → 3. GENERATION/scene_plans/
```

**Добавлять в контекст генерации запрещено:**
- `bible/` — используется при создании scene_plan, не при генерации
- KB карточки — используются только в Критике Б ([2.4](#24--критики--изолированные-агенты))
- `loops.md` — используется при создании scene_plan, не при генерации
- `outline` — уже обобщён в scene_plan

**Почему:** каждый лишний файл в контексте — деградация качества генерации и рост стоимости.
На 500 главах: 1 лишний файл × 500 глав = значимые потери в цене и качестве.

### 2.2 — Canon изменяется только явным решением

`bible/canon.md` — неизменяемый реестр фактов вселенной.

**Добавить в canon разрешено:** только если автор явно сказал «добавить в canon» и подтвердил.
**Изменить canon запрещено:** нельзя «тихо исправить» старую запись. Только добавить `_deprecated` версию и создать новую с audit trail:
```yaml
arkash_eye_color_v1: "тёмно-серые"          # deprecated гл.87, решение автора
arkash_eye_color_v2: "серебристые (глаза Совета после ритуала)"  # гл.87
```

### 2.3 — State — только через structured output

`state_ch_N.yaml` обновляется ТОЛЬКО через JSON schema → YAML конвертацию, не через prose.

**Запрещено:** «Обнови state — Аркаш теперь в тюрьме и его мощь упала».
**Правильно:** вызвать `state_generator` с JSON schema input → получить валидный YAML → проверить `state_validator` ([5.1](#51--валидаторы-запускаются-до-не-после), [5.2](#52--state_validator-проверяет-4-инварианта)).

Причина: prose-генерация галлюцинирует пропущенные поля. `state_validator` не может проверить prose.

### 2.4 — Критики — изолированные агенты

Критик А, Б, В работают в **отдельных контекстах** без общего состояния.

**Критик А:** `draft + outline_ch_N` → структура (G/C/C/C, emotional shift, open loops, хук)
**Критик Б:** `draft + voice_author.md + voice_pov.md + [3-5 KB карточек по scene_type]`
**Критик В:** `draft + canon.md + антипаттерны топ-5` (→ [5.4](#54--антипаттерны--обязательный-чеклист-критика-в))

Критики **не читают друг друга**. Агрегирует человек или отдельный синтетический агент.

### 2.5 — Voice версионируется по расписанию

Голос изменяется каждые **20 глав** — не раньше, не позже:
- Проанализировать `voice_corrections.md`
- Найти систематические паттерны правок
- Создать patch с audit trail: `voice_author.md v1.0 → v1.1`

**Запрещено:** менять `voice_author.md` в произвольный момент после «ой тут нехорошо».
Такие наблюдения → в `voice_corrections.md`. Не в файл голоса.
Smoke test после обновления — обязателен ([5.3](#53--дымовой-тест-голоса-обязателен)).

### 2.6 — Open loops ≤ 5 в активном контексте

`state_ch_N.yaml → open_loops_active: [...]` — не более 5 ID.
Открывать новый loop без закрытия одного из старых — не разрешено.

`state_validator` проверяет это перед каждой генерацией. Если нарушено — генерация не запускается.

## РАЗДЕЛ 3 — AI И LLM ДИСЦИПЛИНА {#раздел-3}

*↑ [К навигации](#навигация-по-документу)*

### 3.1 — Prompt Caching: статика всегда кешируется

`voice_author.md` + `voice_pov.md` + `genre_overlay.md` — статика внутри арки.
Они ВСЕГДА идут первыми в системном промпте с `cache_control: ephemeral`.

Стоимость повторного использования = 10% обычной цены.
На 15 глав арки: 14 из 15 генераций используют кеш. Экономия ~45% от LLM-бюджета.

**Проверка:** в billing логах должен быть `cache_read_input_tokens > 0` начиная со второй главы арки.
Если нет — pipeline сломан. Исправить до следующей генерации.

### 3.2 — Bulk операции через Batch API

Операции над 10+ единицами (карточки KB, extraction, примеры) → **Anthropic Batch API**.
- 50% дешевле
- Без rate limits
- Параллельно

**Запрещено:** генерировать 300 карточек через 300 последовательных API вызовов в реальном времени.
Это в 2 раза дороже и занимает в 10 раз больше времени.

### 3.3 — Structured Output для машиночитаемых данных

State, Canon additions, Loop updates, Bibliography parsing → только через JSON schema.

Prose output → только для: черновиков глав, примеров в карточках, scene plans.

### 3.4 — Модели по ролям

| Задача | Модель | Почему |
|--------|--------|--------|
| Генерация главы | Claude Sonnet 4.6 | Баланс качество/цена на 500 глав |
| Extraction KB карточек | Claude Sonnet 4.6 | Structured output, 300 исследований |
| Примеры карточек | Claude Sonnet 4.6 | Качество примеров критично |
| state_generator | Claude Haiku 4.5 | Простая структура, максимальный объём |
| Критики А, Б, В | Claude Sonnet 4.6 | Качество review важно |
| RLM агенты | Claude Haiku 4.5 | Поиск, не генерация |

Не использовать Opus для рутинных задач. Не использовать Haiku для качественной генерации.

### 3.5 — Fallback обязателен

Каждый LLM вызов в pipeline имеет fallback-поведение когда API недоступен или возвращает ошибку.
Минимальный fallback: сохранить входные данные + залогировать причину остановки + выйти с кодом ≠ 0.

Тихий провал (pipeline «идёт» но без реального результата) — худший из багов.

## РАЗДЕЛ 4 — KNOWLEDGE BASE: ПРОТОКОЛ {#раздел-4}

*↑ [К навигации](#навигация-по-документу)*

Полная логика KB — [0. LIBRARY/LOGICAL.md](0.%20LIBRARY/LOGICAL.md).
Источники исследований — [BIBLIOGRAPHY.md](BIBLIOGRAPHY.md).

### 4.1 — KB не входит в контекст генерации главы

После завершения Step 0 (Bible) и Step 1 (Voice) из [GENERATION_PIPELINE.md](GENERATION_PIPELINE.md), KB **не загружается** в контекст генерации.

KB использовался при создании `bible/` и `voice_*.md`. Его знание уже зашито в эти файлы.
Добавить KB в генерацию = дублирование контекста + рост цены + деградация фокуса LLM.

**Единственное исключение:** Критик Б получает 3-5 релевантных карточек по `scene_type` текущей сцены ([2.4](#24--критики--изолированные-агенты)).

### 4.2 — Поиск по KB только через RLM

Не угадывать. Не пересказывать по памяти. Не «помню, там было что-то про дофамин».

```bash
/rlm "вопрос" --corpus KNOWLEDGE_BASE/cards/ [--filter stage:X scene_type:Y]
```

RLM возвращает конкретные карточки с `card_id`. Ответ строится на основе карточек, не на основе предположений.

### 4.3 — Карточки создаются через LLM

Один запрос = одно исследование из `books/2. Expected/`.
LLM (Sonnet 4.6) определяет домен, разбивает на карточки, форматирует frontmatter и примеры, раскладывает по `studies/<domain>/`.
После — файл перемещается в `books/1. Processed/`, статус в `books/00-queue.md`.

Ручное создание карточек — только для исключений (уникальный источник).
Карточка без `quality: reviewed` — черновик. Критик Б не должен использовать непроверенные карточки.

### 4.4 — BIBLIOGRAPHY.md обновляется автоматически

При каждом сохранении файла в `studies/` — хук запускает `scripts/update_bibliography.py`.
Ручное редактирование BIBLIOGRAPHY.md запрещено — файл перезаписывается скриптом.

## РАЗДЕЛ 5 — КАЧЕСТВО И ВАЛИДАЦИЯ {#раздел-5}

*↑ [К навигации](#навигация-по-документу)*

### 5.1 — Валидаторы запускаются до, не после

`bible_validator` → перед началом Step 2 (Outline) из [GENERATION_PIPELINE.md](GENERATION_PIPELINE.md)
`state_validator` → перед каждым Step 4 (Draft Generation) из [GENERATION_PIPELINE.md](GENERATION_PIPELINE.md)

Если валидатор выдаёт ошибки — генерация не запускается. Исправить → перевалидировать → генерировать.

**Порядок строгий:** validate → generate → critique → commit. Нарушение порядка = undefined behavior.

### 5.2 — state_validator проверяет 4 инварианта

```
1. Все major characters присутствуют в state_ch_N.yaml
2. open_loops_active ≤ 5  (см. правило 2.6)
3. power_level ни одного персонажа не превышает max_power_tier из canon.md
4. last_beat не пустой и содержит ≥ 10 слов
```

Если хотя бы один инвариант нарушен — ошибка с указанием какой именно.

### 5.3 — Дымовой тест голоса обязателен

После каждого обновления `voice_author.md` / `voice_pov.md` (расписание — [2.5](#25--voice-версионируется-по-расписанию)):
1. Сгенерировать 3 абзаца: экшн, диалог, интроспекция
2. Пройти чеклист 5 правил голоса для каждого абзаца
3. Только после прохождения — использовать в реальном pipeline

Не пропускать smoke test «потому что и так понятно».

### 5.4 — Антипаттерны — обязательный чеклист Критика В

Топ-5 антипаттернов (из KB `07_antipatterns`, найти через `/rlm "антипаттерны" --corpus KNOWLEDGE_BASE/cards/`) **всегда** проверяются:
- Weather opening (начало с погоды/природы без нарративной функции)
- Talking heads (диалог без action beats и subtext)
- Info-dump (изложение фактов мира без нарративного повода)
- Head-hop (смена POV внутри сцены без явного разрыва)
- Motivation stated (мотив персонажа объяснён автором напрямую)

Нашёл — указать цитату, объяснить почему это антипаттерн, предложить конкретную правку.

## РАЗДЕЛ 6 — ПРИНЯТИЕ РЕШЕНИЙ {#раздел-6}

*↑ [К навигации](#навигация-по-документу)*

### 6.1 — Горизонт планирования 500 глав

Каждое техническое или нарративное решение оценивается по вопросу:
**«Если это решение распространить на 500 глав — что произойдёт?»**

Примеры:
- «Добавить в контекст генерации Bible целиком» → на главе 50 это будет 80k токенов = сломано
- «Дать герою способность X» → через 100 глав это сделает финальный конфликт нерелевантным
- «Хранить state в prose» → к главе 30 `state_validator` не сможет проверить структуру

Если ответ «не знаю» — решение не принимается до прояснения.

### 6.2 — Script First

Если задача повторяется **2 или более раз** — создать скрипт.

Не «делать вручную пока не много». Второй раз — это момент создания скрипта.

**Скрипты в проекте:**

```
0. LIBRARY/scripts/
  update_bibliography.py    ← авто-пересборка BIBLIOGRAPHY.md (хук PostToolUse)

3. GENERATION/scripts/
  validators.py             ← bible_validator + state_validator
  state_generator.py        ← prose → state_ch_N.yaml (Haiku 4.5)
```

**Правила скриптов:**
- Каждый скрипт живёт в папке слоя к которому относится
- Скрипты запускаются через хуки или вручную — мастер-скрипт не нужен

### 6.3 — ADR для архитектурных решений

Решения которые влияют на pipeline, форматы данных, или выбор инструмента →
оформляются как `ADR-NNNN-название.md` в `.planning/adr/` со статусом `Accepted`.

Минимальные ADR для принятия (согласно [PROJECT.md](PROJECT.md)):
- `ADR-0001`: SvelteKit + Node +server.js
- `ADR-0002`: Filesystem как source of truth
- `ADR-0003`: Qdrant + TEI + BM25 hybrid RAG (или mock fallback)
- `ADR-0004`: /library superseded /journal
- `ADR-0005`: Multi-provider LLM engine

До их оформления — не считать эти решения «зафиксированными».

### 6.4 — Backlog, не «сейчас»

Любая идея которая:
- Не находится в текущем milestone
